# Decision: Roadmap Generation Strategy

**Date**: 2026-02-25
**Phase**: 12, node 1 (integrate-gen-spec)
**Status**: Decided

## Problem

New projects need to adopt roadmap easily. The `executeIntegration()` function discovers metadata but doesn't generate `roadmap.ts`. Two strategies exist:

1. **Bootstrap template** — minimal DAG with init/term, user edits to add phases
2. **Full generation** — auto-generate complete DAG from build process

## Requirements

- Generated roadmap must pass `verify()` without user edits
- Must handle: npm/pnpm, builds, tests, deployment patterns
- Minimum viable: monorepo-services, microservice-deploy, library-ecosystem (adoption scenarios)
- Must be deterministic (same inputs → same output)

## Decision: Hybrid Generation

Generate a **starter DAG** based on discovered build process, with:

1. **INIT** node: existing artifacts (package.json, source)
2. **BUILD** node: output from buildCommand
3. **TERM** node: deployment artifact or dist/
4. Validation: project metadata + build discoverer fully populate the DAG

### Why not pure bootstrap?

Bootstrap leaves user with:
```typescript
{ id: 'init', ... }, { id: 'build', ... }, { id: 'term', ... }
```
Then they must add 10+ nodes manually. High friction.

### Why not pure generation?

Pure generation would require:
- Inference of deployment strategy
- Assumption about what "done" means (unsustainable)

Hybrid: **Auto-generate init → build → term**, then **agents expand with custom phases**.

## Implementation

1. `generateInitNode(metadata)` — produces: source files from package.json
2. `generateBuildNode(buildProcess, metadata)` — produces: build output
3. `generateTermNode(buildOutput)` — produces: final artifact (dist/, lib/, etc.)
4. `validateGeneratedDAG(dag)` — verify(), check(), all succeed
5. Constraint: all nodes marked `idempotent: true` (agents can re-run safely)

## Generated Roadmap Structure

```typescript
export default define(graph({
  id: 'my-project',
  desc: 'Project roadmap (auto-generated by roadmap integrate)',
  init: 'init',
  term: 'term',
  nodes: {
    init: {
      id: 'init',
      desc: 'Initialize project: install dependencies, setup',
      produces: ['node_modules', '.git/hooks'],
      consumes: [],
      deps: [],
      idempotent: true,
    },
    build: {
      id: 'build',
      desc: `Build project: ${buildCommand}`,
      produces: ['dist', 'lib', '.d.ts files'],
      consumes: ['src', 'tsconfig.json', 'package.json'],
      deps: ['init'],
      idempotent: true,
    },
    term: {
      id: 'term',
      desc: 'Project ready for deployment',
      produces: [],
      consumes: ['dist', '.d.ts files'],
      deps: ['build'],
      idempotent: false,
    },
  },
}));
```

Agents can then:
```
orient(dag) → { position: 'build', produces: [...] }
// Create the artifacts, commit
orient(dag) → { position: 'term', ... }
// All phases done
```

## Validation

- Minimal 3-node DAG passes all tests
- Doesn't invent phases (no deployment, no testing, no staging)
- Users can add phases via `reconcile()` as needed
- `generate-roadmap.test.ts` covers: npm, pnpm, monorepo, monorepo-build patterns

## Rejected Alternative

**Option A**: Ask user "what's your deployment target?" → inference-based generation
- Rejected: Users don't know how to answer, too many edge cases

**Option B**: Full scaffolding (init, lint, test, build, deploy, term)
- Rejected: Assumes deployment model, fails for libraries that don't deploy
- Later phases can add these via DAG merge

## Success Criteria

- `roadmap integrate --dry-run` outputs valid roadmap.ts
- Generated DAG passes `verify()`, `check()`, all validation rules
- CLI succeeds without user edits
- Works for: npm, pnpm, yarn, monorepos (root + packages)
